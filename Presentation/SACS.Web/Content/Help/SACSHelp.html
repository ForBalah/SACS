What is it

SACS is a centralized Windows service manager. It draws inspiration from Windows Task Scheduler, Windows services, and SQL jobs, combining their concepts into a single package. It is designed to help developers, testers, and support engineers expedite the building, validation, deployment, and maintenance of Windows-like services.

The premise of SACS banks on improving the reliability of existing Windows services, drastically cutting down on development time by taking over the boilerplate, peripheral, and ancillary items which are required when setting up a new Windows service. These items include, but are not limited to: allowing multiple instances of the same service on the same system, error logging, notifications, scheduling, and enabling analytics.

More than just a job scheduler

Although SACS can execute your services at any to-the-minute interval you throw at it, you can just as easily not specify a job schedule and instead have your service remain in an always-on state. This is especially important if your service will contain listeners that dont need to be on a timer (like a file system watcher or a message queue broker).

Your services are setup up as simple executables which can be configured to run in a multitude of ways with the help of SACS.

A brief look at the innards of SACS

First, a Service App (SA) is created. This is the executable that is written in an app-like fashion, with the intention for it to be deployed as a service. These service apps are not restricted to running in SACS they are plain old .NET console applications with the added feature of being able to send and receive communications that SACS understands. Testing the service app is as simple as running it as a normal console application and providing it with commands in the same way that SACS would instruct it. (See Creating a Service App for more information on what instructions a service app understands). At this point, nothing other than the payload of the service app is developed and tested. No more need to worry about timers, error logging, support emailing

The service app is then deployed to a Service App Container (SAC). These will usually reside on the same server. Service app deployment typically involves manually copying, or using an automated file drop service, to place the service app files on the same server as the SAC and then using a Windows management console to load the service app into the container.

The last S in SACS is the Server which combines all the components, such as the container, into a cohesive system.

Server Components

Along with the container described above, the server is also made up of a few other core components:

SACS Agent This is the Windows service handler for the server. It controls the starting, stopping, and restarting of SACS as a Windows service (the only Windows service you will need from this point on), including all its components.

Scheduler This manages the timings for all service apps that have been configured to run according to a schedule. The container relies on the scheduler to tell it when to execute service apps.

System monitor When active, this component constantly monitors the health of SACS; keeping track of key CPU and memory metrics across all the service apps hosted in the containers.

Communicator This component is the gate between the server and the various management consoles that communicate with it.

Wizard Installer

The setup wizard is the easiest way to get SACS installed onto a target system. All required files and deployment scripts are bundled into the installer so no further downloads are necessary. Only the prerequisite .NET and SQL Server versions are required before proceeding with the setup.

To run the wizard, locate and execute the SACS.Setup.exe.

Note: Administrator privileges will be required on the target machine in order to run the setup.

Refer to the Manual Installation section if you opt to not use the wizard to perform installations/upgrades. The Wizard can still be used to perform upgrades after a manual installation.

The different Setup stages

The wizard offers 3 setup stages:

1.     Install / Upgrade For new installations and for upgrading existing installations. Note the version number of the wizard (displayed in the title bar). This represents the version of the system that will potentially be installed onto the target machine. Once this section is complete, the wizard automatically moves to the configure section.

2.     Configure This is used to make any tweaks to the configuration settings of the server and management consoles, as well as deploying any database updates that are required.

3.     Uninstall This will identify any known installs of SACS and delete those instances. Any existing service apps on the machine will remain untouched.

Figure 1: Example of what updates to the server and Windows console would look like

Server and Container Installation

Installation of the server is the main requirement for SACS. The wizard identifies an existing installation by searching for the installed service agent on the target machine. If it can find it, the location of the agent is used and it cannot be changed without first uninstalling the existing instance.

Note: The setup wizard will automatically back up any files to a SACS_Backup folder before applying any updates to existing files. This folder sits at the same directory level as the SACS system.

On clicking the Install Now or Update Now button and passing the prompt checks, the following will take place:

1.     The service will be stopped automatically if it is already installed. It will not be started again from within the wizard use a Windows management console to start the service again.

2.     Existing files will be backed up into a zip file called SACS.WindowsService <date>.zip.

3.     The file version from the installer will be copied over to the destination, ignoring any custom configuration files.

4.     Once installed, the entire section will be disabled and an Installation Complete message will be displayed.

This section of the installation can be skipped for any reason. To do so, check skip this part. This will disable the section and no further actions are possible.

Windows Management Console Installation

The setup wizard will first search for an existing installation of the console by checking the start programs for a shortcut that references an existing Windows management console executable. If it finds an existing location, that will be used for installations and cannot be changed without first removing the shortcut or uninstalling the existing management console.

Worth noting is that locations containing existing Windows management console files can be used. You will be prompted that the files will be upgraded in this location and will be used as the install location. During the installation a shortcut will be added to the start menu programs which allows both ease of access to the console, as well as enables the wizard to locate the primary Windows management console installation.

As with the server and container installation, this section of the installation can be skipped for any reason.

Note: The setup wizard will automatically back up any files to a SACS_Backup folder before applying any updates to existing files. This folder sits on the same directory level as the SACS system.

Configuring the Server and Container

Figure 2: The configuration screen for the server and container.

The server and container configuration is broken down into 3 sections:

1.    Service Account configuration

This is the account the SACS server will be starting up as. The refresh button will show any updates to the services account. Clicking on Open Services will attempt to open the service properties on Windows so that the account can be changed. This does not always work so if the main services window is the only window shown, search for the SACS Agent service and open its properties.

Note: There are two possible options for service accounts. Using LocalSystem by default has all the privileges needed to run SACS but is unable to access remote SQL databases easily. A separate service account is preferred, ensure that the following rights are assigned to it:

         Log on as a batch job

         Log on as a service

         Adjust memory quotas for a process

         Replace a process level token

If the database will be installed to a different server, use an account which has limited read/write rights to a SACS database and can also be made as an administrator on the machine with the server. Refer to Troubleshooting SACS if there are any issues starting the service.

2.    Database deployment

SACS requires a SQL database in order to start up. A local database file is included with each installation of SACS, however if there are schema changes to the database, or if the database cannot run on the same server as SACS (e.g. if SQL Server is not installed) then this section can be used to install or upgrade the database. Click on Deploy Scripts to kick off the database upgrade.

3.    Configuration file

The App.config section contains all settings that are modifiable for the server and container. This includes app settings, connection strings and SMTP settings. Refer to Configuring the Server for any specific configuration issues.

Note: Any changes to the config file are first backed up (to a file with a backup date at the end) before the changes are applied. Restoring a recent back will require opening up the location of the SACS server and restoring the contents to SACS.WindowsService.exe.config file from the required backup which will be in the same location.

Remember to periodically clean out the backup files if multiple changes to the config are made over an extended period of time.

Using SACS

Step 1: Create a Service App

SACS aims to make getting started with services incredibly easy. This is allows the focus to be turned to what the service will do and not have to consider all the various typical boilerplate tasks required to get a service running since SACS will take care of that for you. Refer to the Service App Development section under Developer Notes for a tutorial on how to create a service app.

Step 2: Deploy the Service App

Once a service app has been developed and tested, it can be deployed to a SACS container (with a default instance residing on the SACS server after installation). Usually the files will sit on the same Windows server so that SACS can easily control the start/stop/execution of the services.

A Windows management console is required to deploy a Service App.

To deploy a service app:

1.     Copy the compiled service app executable and supporting files to the machine with SACS installed.

2.     Open a Windows management console on that server configured to point to SACS.

3.     The console will make various attempts to contact the server. If the server is offline, start the server and refresh the sections. On successful connection, navigate to the Application Management tab and click on Add.

4.     On the right hand side, the details of the service app can be entered, including the name, location and schedule it should run on.

5.     Click save. If successful, the service will appear on the left-hand side.

The Windows Management Console section contains more detailed instructions on how to deploy a service app.

Step 3: Manage the Service Apps on the server

With the service app now deployed to SACS, from here it is possible to start, stop, and run the service app. Every action performed on a service is logged and is available for viewing. Performance data is also saved, although it is not accessible directly from the console and requires a third-party analytics tool to view.

Should there be an issue in a service and it crashes during execution, an email will be automatically generated and sent to the support email address defined in the servers app.config section as per the wizard installation.

Windows Management Console

The Windows management console is the primary means of communicating with a SACS server and provides all of the features required to manage the server and service apps. Its primary features include:

1.     Managing the SACS server. This involves:

a.     Stopping and starting the server (through the SACS agent)

b.    Accessing the logs which are written to the server

c.     Viewing the latest live CPU and memory performance graphs of the server

d.    Viewing the configuration settings that both the server and the console contain

2.     Managing service apps by:

a.     Facilitating the deploying and removal of service apps from the SACS container.

b.    Editing the deployment details of service apps, from their location, start up type and execution schedule.

Managing the Server

Figure 4: Example of the Windows Management Console

On opening the Windows management console, the first view will be the Server Management tab. The console will make multiple attempts to connect to the server it is configured to connect to. If the server is not running, various error messages will display indicating that there was an issue reaching the server.

Note: Almost all activity in SACS originates from, or is processed by the server. So the Windows console has minimal functions when the server is offline. The only items available when the server is offline is: starting the server, and viewing the logs (when their alternate location has been specified).

 

Starting the server

Click on the Start Server button. This will send a command to the agent to commence the start-up sequence. The console will update with Start up in progress. Clicking on the refresh button will fetch the latest status of the start-up process. Bear in mind that it will take upwards of 8 seconds to start the server. Once the server is started, the service apps which were deployed to the container will spin up in sequence.

Stopping the server performs the process in reverse and is kicked off by clicking on Stop Server. Each service app is given 15 seconds to finalize its processing and exit before it is force-stopped by the server.

View performance information

There are two graphs available on the right-hand side of the Server Management tab: CPU usage and memory usage.

The CPU usage graph shows what the total usage of SACS and its service apps are at a point in time. SACS periodically checks the health of the service apps and CPU usage may spike slightly as log records are requested. Otherwise it is incredibly rare for the server to have high CPU usage (more than 1% usage). The CPU usage of service apps is collated and displayed on the same graph.

The memory graph shows the memory footprint of SACS and its service apps at a point in time. SACS typically consumes between 20-50 MB when logs have not been requested, and 50-100 MB when it filters and serves log records. This drops back down to 20-50 MB after a few minutes.

The default look-back time for the graphs is 2 days. This is configurable for both graphs in the app.config settings for the console and can be done by the setup wizard. The graphs record their data every one minute, on the minute, and they will refresh every minute thereafter. A refresh link is available at the bottom of the graphs which will fetch the latest graph information immediately.

View configuration information

Each management console displays configuration related to both its setup and the servers setup. This is viewable at the bottom of the Server Management tab by expanding the Configuration section. The details in this configuration are not editable from within the console. Use the setup wizard to make any amendments to an existing installation of either the server or the Windows management console.

Logs

The list of logs are sourced and served through the SACS server. In the event that the server is offline, the Windows console will perform a physical file search for the logs using the Alternative Log Location path specified in the app.config section for the console.

All logs are physical files in an XML format, with the default maximum log size being 4 MB. Once a log rolls over that size, it will be backed up to the name of the log with the log number at the end. E.g. SACS_logs.xml.1. These logs are still viewable by the log viewer and all logs that reside in the configured location will be displayed in the list. There is no mechanism in place to automatically clean out old log files so be sure to clean out or archive old logs periodically.

Log Viewer

Selecting a log and clicking on the Open Log button will open the Log Viewer.

Figure 5: Example logs inside the log viewer

The log viewer displays the logs in an easy to read format which can be filtered, sorted, and searched on.

Figure 6: The Filter logs dialog

Clicking on the Filter menu will bring up the Filter Logs dialog where it is currently possible to filter logs my username, level and message. Clicking on OK will apply the filter.

To clear the filter, either close and reopen then log viewer, or click on the filter button again and click OK while leaving all the fields blank.

The Search in the top right section will currently search the Level and Message columns in the log for the specified text. The log search occurs from the selected log record downwards. If the end of the log is reached, a message is displayed indicating that the search has stopped.

When selecting a log to view the details, the fields displayed at the bottom half are as follows:

         Item The log item number within the specific log.

         Line The line number in the source file where the log message originated.

         File The location of the source file that contained the code that produced the log message.

         Timestamp The time, in UTC when the error occurred. This is translated into the local time zone before display.

         Hostname The name of the server where the log message originated from.

         Method The method within the source file where the message originated.

         Machine Name The name of the machine where the log message originated from.

         Username The username, typically that of the SACS server, which produced the log entry.

         Class The class within the source file where the message originated from.

         Thread The thread ID that was used to process the code which resulted in the log entry.

         App The name of the app that created the log entry. This will typically be the SACS Windows service as it logs errors and messages on behalf of the services that it starts and runs.

The two tabs at the bottom give a detailed account of the log record:

         Message Shows any message that was recorded along with the log entry.

         Throwable Shows the full stack trace that occurred, if there was an error that was logged. This tab is only viewable when clicking on a log entry to view its details.

Note: The SACS server also emits debug information. However it is filtered out from the logs by default. To add debugging information to the logs, manually edit the SACS.WindowsService.exe.config file for the server, locate the logging section, and change the minimal log level to DEBUG. Be warned that a lot of data is sent to the logs in debug mode and should only be used in cases where a particular issue requires verbose information to diagnose.

 

Managing Service Apps

Figure 7: The service app list

Click on the Application Management tab to view and administer service apps. From this tab it is possible to refresh service apps, add new service apps, remove existing service apps (only from the container, not from the system), export the service app list, and control service apps remotely.

The Refresh button will contact the server and fetch the current status of its service apps.

The Add button will kick off the process of adding (deploying) a service app.

The Export button will export the existing list of services to a csv or a tab delimited file.

The Metrics button is a work in progress. In the meantime, third party analytics tools can be connected to the SACS database to enable that functionality.

The Help button will open the current help manual.

Note: If the SACS server is offline this section will not be available. The Windows management console will only check server availability when clicking on the Application Management tab.

 

Viewing current service apps

The list on the left-hand side shows the service apps currently deployed to the server. The columns are as follows:

          [Status image]. This shows the current status of the service app. The current list of statuses are:

o   - The service app is stopped. In this state it can be changed or removed.

o   - The service app has a new message and the list should be refreshed to see the changes.

o   - The service app has started successfully and is set to run automatically according to its prescribed schedule.

o   - The service app has started successfully but is set to manual execution. Therefore, despite the schedule assigned to it, it will not run unless it is manually triggered using the Run immediately button in the service app details.

o   - The service app is started, but there was a problem detected during start-up. The state of the service app cannot be tracked and no commands, other than stop can be processed by the app. This usually occurs when the application loaded into the container could not be recognized as a service app.

Since service apps are plain old console applications, which happen to contain a specific library, when SACS starts a service app, it kicks off a handshake process with the application to determine if it is indeed a service app and gets its version and capabilities. If any part of the handshake process fails then this warning sign is displayed.

o   - The service app has errored out during execution. The last message is shown on the right-hand side under the Last message section or by viewing the logs. An email will be sent to the configured support email when this occurs.

o   - The service app has been disabled and cannot accept any further instructions. To take a service app out of a disabled state, click on the service app and at the bottom of the details, click Edit and change the start-up type.

o   - The state of the service app is unknown as some its details have been altered. Please refresh the list to see the updated state.

         Name, Environment, Description

o   These three columns describe the service app. The Name column is a unique identifier for the service app. The Environment column shows what grouping it belongs to. The Description is a short description of the service app.

         Startup Type

o   Automatic means the service app will run according to the prescribed schedule

o   Manual means the service app will only run when manually triggered

o   Disabled means the service app will not start and will not receive any commands.

         App version

o   Displays the product version of the service app.

         Send Success Notification

o   If true, then every time a service app completes execution, it will send an email to the support email setup in the app.config file for the server.

See Creating a Service App on how to create a service app.

Service app details, deployment and manipulation

The service app details section on the right-hand side displays all extra information related to a selected service app, allows the manipulation of a service app, and also allows editing and deployment of service apps

Note: The service app details are read-only unless the Edit button at the bottom of the details is clicked. Only when the service is stopped or disabled, is it possible edit an existing service app.

Figure 8: Example existing service app. For a new service app, the "service App Name" at the top would be editable.

Clicking on the Add button will start the process of deploying a service app into the container. Make sure that the files are already present on the machine before proceeding.

The various sections of the service app detail are:

         Service App Name:

o   This is the unique identifier of the service app. Once the service app is created, it cannot be altered.

         Last Message:

o   Displays the last activity message related to the selected service app.

         Start, Stop, Run Immediately buttons:

o   These buttons allow control over the service app.

o   The buttons will be disabled when editing a service app.

o   Start Sends the command to start the service app, if it is not in a disabled state.

o   Stop Sends the command to stops the service app. If the service app is busy processing, the service is given 15 seconds to quit before it is force stopped by the server.

o   Run Immediately Sends the command to run immediately, despite the schedule given to the service. This can be fired as many times in succession as possible, however, based on the execution mode that has been setup within the service app, the multiple commands may, or may not be fulfilled. See Service App Development for further information on the execution modes that affect a service app.

         Startup Type:

o   Determines the start-up mode for the service app when SACS is started. The types are

o   Automatic The service app will start up when the SACS server starts up and it will automatically start running according to its prescribed schedule.

o   Manual The service app will start up when the SACS server starts up, but it will not run on a schedule. Its execution must be manually triggered by clicking on the Run Immediately button after the service has been started.

o   Disabled The service app will not start up when the SACS server starts up. Moreover, it cannot be started unless it is taken out of disabled mode.

         Environment:

o   This is the name of the group that the service app belongs to. Typical values include: Development, Testing, Staging, Production, but this accepts any text.

         Description:

o   This is a short description of the service app.

         App File:

o   This is the location of the service apps .exe file. This file should reside on the same server as SACS and it should have all necessary files and DLLs installed in order to function similar to if it were executed as a normal console application.

Notice that no config file needs to be specified, once again, each service app is treated as a console application but with added communication features therefore any required config files should be bundled with the service app being deployed.

         Notifications:

o   The Send Success Notification checkbox can be checked so that emails are sent to the support box on successful execution of a service. This is included for both automatic and manually triggered runs.

         Identity:

o   This is the user account that the service will run under. Clicking on Choose will open the Select Account dialog where the SACS account or a separate, custom account can be selected. The account password is never stored on the Windows console. After entering the account details, the password is immediately encrypted and is deleted once transferred to the server.

Figure 9: Example Identity selection box

o   The identity can be left as the built-in account which will allow the service app to run under the SACS account, or it can be entered so that the service will start and run under a different context from the SACS account.

         Schedule:

o   This defines the schedule that service apps in an Automatic start-up type will follow. The details shown on this section are written in a natural language format. However it is not possible to specify natural language to the scheduler. Click on the Choose button to open the schedule selection window.

o   The schedule dialog allows for the configuring of schedules in 5 parts: minutes, hours, days, months, and days of the week.

o   The calendar and hour/minute break down within the description section allow you to select a day and visualize what the run times will look like.

Figure 10: The schedule selection dialog.

The various parts of the schedule dialog are explained as follows:

Time - Hours

Time - Minutes

At: allows multiple minutes to be selected. The schedule will be updated with the minutes selected accordingly.

Every: allows one minute to be selected. If All, 0 or 1 are selected, the schedule assumes it will run for all minutes within the prescribed hours. Selecting any single number afterwards will imply the schedule runs at that interval. Multiple minute selections are not allowed with this option.

Day

Use this to specify the days within the designated months that a service app will run. The dropdown contains the values 1-31, representing the day of the month. 31 represents the last day of the month.

At: allows multiple days to be selected. The schedule will be updated with the days selected accordingly. Note that on that day, all prescribed hours and minutes will be active.

Every: allows one day to be selected. If All or 1 are selected, the schedule assumes it will run for all days within the prescribed months. Selecting any single number afterwards will imply the schedule runs at that interval. Multiple day selections are not allowed with this option.

Month

Use this to specify the months within a calendar year a service app will run. The dropdown contains the values January to December.

Every: allows one month to be selected. If All or January are selected, the schedule assumes it will run for all days within the prescribed months. Multiple month selections are not allowed with this option.

Use this to specify the particular days of the week to run the service app on.

This is an intersection therefore if the prescribed day within the prescribed month falls on the selected day of the week the will schedule apply.

This shows both a natural language selection of the schedule as well as a calendar with an hour and minute visual breakdown of when the schedule will execute. Blue lines indicate the time when the schedule applies.

Validation

If there are any issues with the schedule selection, the messages will display here. The schedule will only save once the validation errors are resolved.

o   Clicking OK will apply the schedule to the service app

         Configure

o   This is where the Save, Remove, Cancel, and Edit buttons will be available.

o   Save Will validate and save the current service app changes.

o   Cancel Will undo any current changes and cancel the edit view.

o   Remove Will remove the service app from the SACS container. This will not remove the service app files from the system

o   Edit Will kick off editing of a service app once it has been stopped.

 

Web Management Console

The web management console is a slimmed down version of the Windows management console. It is designed for remote administration of deployed service apps. Currently it has to be installed manually. See Manual Web Management Console Installation for further installation information.

Figure 11: Example main dashboard page of the Web management console

Server Information

Web Dashboard

Once the web console has been installed, navigate to it. The Dashboard page is the default page that is displayed.

Like its Windows management console counterpart, the dashboard displays the list of logs, CPU and memory performance, and configuration details of both the server and the web console.

Unlike its Windows counterpart, the web console does not display the server details as currently it has not be built to administer the server remotely.

Logs

Clicking on the view button against a particular log will open up its details, much like the windows log viewer. Clicking on the Logs link on the left-hand side will expand the list of logs on a new page.

As with the windows console, the location of the logs is controlled by the SACS server and all log requests go via that server. However, if the server is offline, the logs will be inaccessible as there is currently no mechanism built into the web project to probe a remote machine for the logs.

Service App Web Management

Clicking on the Service Apps link on the menu will load the list of service apps currently loaded into the SACS container that the web console is pointing to.

Figure 12: Example service app list in the web console.

Similar to the Windows management console, the web console will display the current status of all the service apps in the container. Clicking on the refresh button link will fetch any updated status.

The buttons function similar to the Windows console counterpart.

         Start, if enabled, will send a command to the server to start the targeted service app. A popup will display indicating that the command was sent successfully. The Refresh button can be clicked to receive updates on the status of the service app.

         Stop, if enabled, will send a command to the server to stop the targeted service app. A popup will display indicating that the command was sent successfully. The Refresh button can be clicked to receive updates on the status of the service app.

         Run, if enabled, will send a command to the server to run the targeted service app immediately. A popup will display indicating that the command was sent successfully.          Log on as a batch job

         Log on as a service

         Adjust memory quotas for a process

         Replace a process level token

Tutorial: Creating a Service App

Prerequisites

This tutorial assumes SACS is installed and running on localhost on the default port (3800). In case you use a different host, port or credentials, connections settings would require adjusting.

Where to get help

If you're having trouble going through this tutorial you can contact us through the mailing list.

 

Show me the code

Service Apps are designed to be built on top of Windows console applications. To create a new, basic service app:

1.     Create a new .NET 4.5, or above, Windows Desktop Console Application.

2.     Locate the SACS.Setup folder. Within the folder there is a lib folder containing SACS.Implementation.dll. Add this dll to your solution and add a reference to it in the new project.

3.     Add this code to the Program.cs file:

public class Proclaimer : SACS.Implementation.ServiceAppBase // 1

{

private static string greetings;

 

protected override void Initialize()

{

greetings = "Hello World"; // 2

}

 

public override void Execute(ref SACS.Implementation.Execution.ServiceAppContext context)

{

Console.WriteLine(greetings); // 3

}

 

public void StartServiceApp()

{

this.Start(SACS.Implementation.Execution.ExecutionMode.Idempotent); // 4

}

}

 

public class Program

{

private static void Main(string[] args)

{

Proclaimer p = new Proclaimer(); // 5

p.StartServiceApp(); // 6

}

}

 

4.     Build the solution

5.     Right-click on the console project, hover over Debug, and select Start new Instance

You should see an output window that looks something like this:

6.     To run the service app and see it in action, type the following and press enter

action: "run"

The results will look like the following. Notice the Hello World text in the middle.

7.     To exit the service app, type exit (without quotes) and press enter.

The breakdown

(Referring to the comments in the code in Show me the code):

1.     Our program needs to implement ServiceAppBase as the first part in turning it into a service app. This can easily be added onto a separate class, in a separate project as long as it has the right references.

2.     In the Initialize method, anything related to the service app gets initialized. In this case it is the message to be proclaimed.

3.     In the Execute override, the run action of the service app is performed. Here the Proclaimer proclaims the message.

4.     A method is created to access the protected ServiceAppBase.Start() method. A service app can be started in one of three execution modes through which it dictates how often the call to execute can be fulfilled.

5.     Here we create a new instance of Proclaimer which will perform our service app actions.

6.     The process of turning our console application into a service app begins with the final call to the proclaimer to start.

Points to ponder

One way of making sure that the SACS.Implementation dll is the same across all service apps on a server is to add it to the GAC and reference it from there.

Components, like authentication, and anything else that can be done in a normal console app, can be done in the service app.

All plain messages written to the Console within a service app are recorded in SACS as info log entries. So use Console.WriteLine to log event information.

Any unhandled exceptions from a service app are re-packaged and sent to SACS for logging before the app attempts to recover from the error.

Service App Debugging and Testing

Debugging and testing the service app can be done directly as a console application whether it has been deployed already or not. The service app emits JSON which is designed to be consumed by a SACS container.

To debug a service app, either:

1.     Run the console exe and attach a debug process to it, or

2.     Start a debug session from directly in Visual Studio using the Debug Start new instance menu (recommended)

In either case, when the console application starts up, only the Initialize method is called. To call the Execute method, a run action command will have to be typed in the console (as such action: "run"). There are other commands available and are explained below.

Service App Commands

The following commands can be typed in the console to perform different actions on a service app to test it. They are:

Command

Effect

action: "run"

Instructs the service app to queue an execution. The service app will itself figure out if it can fulfil the command (this is based on ExecutionMode).

For example, if the ExecutionMode is set to Idempotent, and the service app takes 1 minute to execute, any run commands sent within a space of 1 minute when the service app is already running are ignored. For Concurrent mode, a new execution will be queued every second for each run call.

action: "stop"

Instructs the service app to shut down. It will attempt to gracefully free its resources before shutting down. In an event of a failed shutdown, the application will have to be closed manually. In SACS, once a service app has received the stop command, it has a default of 10 seconds to shut down gracefully before it is force closed.

info: "appVersion"

Returns the version of the implemented service app.

info: "sacsVersion"

Returns the version of the SACS.Implementation dll that the service app is using.

exit

This is a shortcut for action: "stop", instructing the service app to shut down.

 

Configuring a Service App

Since service apps are console applications at their heart, the behaviour of a configuration file within the application does not change.

There are certain configuration settings that are used by SACS.Implementation that can alter the service app behaviour. These settings can be added to the configuration file and are all optional. The various settings are detailed below:

Name and Location

Description

Values

Log activity to file

appSettings/add[@key=SACS:LogToFile]

Indicates that activity from within the service app should be logged to a file. Activity that is logged: starts/stops/executions, receiving commands, error handling, sending data to SACS.

Default: false

true/false are accepted.

Maximum number of concurrent executions

appSettings/add[@key=SACS:MaxConcurrentExecutions]

The number of executions of a service app that are allowed concurrently when the execution mode is set to Concurrent

Default: 5.

Any value 0 or greater is acceptable. A value of 0 indicates no internal concurrency limit and is inherently determined by the number of threads available to the process.

Service App alternate name

The alternate name of the service app as seen from within itself. SACS can assign a service app a name that the server will refer to, but a different moniker can be used inside the app to give control over its internal usage.

Default: the service apps assembly name.

Any non-empty name is acceptable.

A custom string containing any parameters that the service app relies on.

Service App Deployment

A deployment of a service app involves these steps:

1.     If there is an existing service app deployed, make sure that it is not started. This can be done either via any Windows or web management console that is pointing to the correct server, or making sure that the service apps process is not running (this requires logging onto the machine).

2.     Copy the compiled files over to the destination of the deployment, making sure to backup existing files in the process.

3.     Perform the steps outlined in Step 2: Deploy the Service App to deploy the service to the container.

Deployment Fall-back

In an event where service apps cannot be deployed to a server container, there are still a number of ways in which service apps can be deployed.

Note: Although service apps are designed to mimic Windows services, they cannot be deployed as them. This is because service apps do not implement the required service interfaces that enable them to be deployed as Windows services.

 

Because service apps are console applications which can also receive command line arguments on start-up, the following methods are available to deploy service apps outside of a SACS server:

1.     Run the console application as standalone program. Use this method if the service app is typically only run once on start-up. To add it to start-up, create a shortcut which points to the executable.

a.       Open the shortcuts properties and in the target add the command flags that would be required of the service. To add multiple commands, surround them in the JSON array syntax. Only one of each command is allowed. E.g.

{ action: ["run", "stop"] }

b.      Add the shortcut to the Startup folder in the start menu so that it starts with the system reboot.

2.     Run as a Windows Scheduled Task. Use this method to run the service app at set intervals. The service app can be setup as follows:

a.       Open Task Scheduler and click on Create Basic Task

b.      Enter the name and description and click next

c.       Select when you want the task to run

d.      Choose start a program

e.       Enter the path the executable. Add arguments to the arguments box, remembering that the commands are treated as JSON

f.        Click Next and Finish.

When setting up a service app using either of these methods, some features will become unavailable such as email notifications and logging. This is because the SACS server contains this functionality.

Troubleshooting SACS

This section details known issues, along with possible remedies, that could occur when installing, or running SACS.

Wizard Installation issues

Sometimes there will be issues running the installer. Most problems will occur due to insufficient rights. Here are some common problems and how to solve them.

When running the setup wizard, all installation steps are saved to a log file. It is available at %TEMP%\SACS.Setup\InstallLog.txt

Access Denied when selecting the Install/Update option

Possible causes:

-      The setup wizard searches the start menu programs as part of arranging an installation/upgrade. The user may have insufficient rights to view all the start menu programs.

Remedies:

-      Be sure to be an administrator on the system with the rights to view all start menu programs.

Server Start Issues

Most server start issues originate from database connection being severed and service account not having sufficient rights. Here are some server start problems and how to solve them:

Access Denied on start

Possible causes:

-      The account that the SACS server is running as does not have access to the database.

-      SQL server is not installed on the target machine.

-      The local MDF file was created with a different login

-      If attempted from the Windows management console, the logged in user has not started the console with administrator privileges.

Remedies:

-      Check that the account that accesses the SACS database has sufficient rights to log in and read/write records.

-      Make sure SQL server is installed on the target machine and that the server is configured to point to it.

-      If the local MDF file that comes with SACS is used, then bear in mind that only one user can log onto that database at the same time. So make sure that SACS is the only login accessing the database.

If there are access issues on an existing database, it means that a different login is claiming rights to the database, this gets handled in the LDF file that accompanies the MDF file. Delete this file and they try to start the server again.

-      A red symbol should appear next to the server status if the logged in user does not start he Windows console with administrator privileges. To remedy, the user should either start the Windows console as an administrator, or ensure that they have logged on the machine with full administrator rights.

Agent start timeout after 30 seconds

Possible causes:

-      Connection to the database has deadlocked and may have timed out.

Remedies:

-      Open up the latest log file (found at <SACS.Service location>\Logs\SACS_logs.xml) and view the latest message. If it states that there was a timeout, try restarting the agent. If restarting does not work, check if there are any deadlocks on the SACS tables in the database and resolve and discrepancies. Restart the service agent afterwards.

Server is hanging on start

Possible causes:

-      The provided MDF file is corrupt.

Remedies:

-      Reinstall SACS or configure the server to use a different database that resides on an accessible SQL server. Be sure to check that the account SACS is using has access to that database.

Windows Management Console Issues

There was a problem loading the state of the service Error

Possible causes:

-      If this error is accompanied by Invalid URI errors underneath the logs, it may be a configuration issue.

Remedies:

-      Expand the Configuration section at the bottom of the window.

o   If each entry is empty, it means that the config file was not copied over during installation. Reinstall the latest version of SACS to resolve the issue

o   If the Configuration section is not empty make sure that the user has administrator privileges to SACS.

Error Executing GET for resource path Error under logs or on Application window.

Possible causes:

-      The server has been stopped.

-      The console is pointing to the wrong server URL.

-      The server URL is inaccessible.

Remedies:

-      Check that the SACS server is running. Do this by seeing if the SACS Agent has started.

-      Check the configuration files of both the SACS server and the console to make sure that they have the same address.

-      Check that the port that the server is on is accessible to the console.

Web Management Console Issues

Failed to load xxx

Possible causes:

-      This is a generic message showing that no connection to the SACS server succeeded.

Remedies:

-      Check the web server log to see details of the error. Or use browser dev tools to monitor the network requests and see the underlying failure.

Usually the server is offline. If the server is online, check the configuration settings on both the server and the console to make sure that the base URL is the same.

If there are still issues, check that the port is open and that the console application can ping it.

Restart SACS server after making any change.

Unauthorized 401

Possible causes:

-      Windows authentication is not enabled for the website.

-      The user does not have access to the web management console.

Remedies:

-      Within IIS, check that the Authentication type is set to Windows. Under Providers, the typical order of providers is Negotiate, followed by NTLM. If this order is different, check with your network administrator to confirm that the choices are correct. If there is still a log in failure, restart IIS.

-      The user must be granted access to the application via Windows AD.

Service App Issues

Access denied

Possible causes:

-      The service account that SACS is running under does not have rights to the folder where the service app resides.

Remedies:

-      If not using the LocalSystem account for SACS, check that the configured service account has been granted at least read rights to the folder where the service app resides.

and message starting service app shown indefinitely

Possible causes:

-      The exe that was loaded is not a service app. or

-      The exe process crashed before it started

Remedies:

-      Check that code is correctly setup to start as a service app. See Tutorial: Creating a Service App to confirm what the code layout should be.

-      The process should show in task manager. If it is not present, it means that it exited before it started and SACS was not informed. The errors can usually be found in the Windows event log, focussing in the System log.
